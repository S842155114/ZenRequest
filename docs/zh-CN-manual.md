## 第一次打开后先做什么

ZenRequest 是一个**本地优先、轻量、离线优先**的桌面 API 工作台。它不是围绕账号、云同步和团队协同设计的，而是围绕开发者在本机上高频进行接口调试、问题复现和协议验证的真实工作流设计的。

第一次打开后，建议先确认三个区域：

- **工作区与环境区域**：用于切换当前工作区和环境变量上下文
- **请求工作台**：用于创建 HTTP 或 MCP 请求
- **设置与帮助入口**：位于顶部设置区域，可快速进入应用内帮助对应的仓库文档落点

如果你只是想先确认应用能正常工作，最简单的路径是先发送一个 HTTP 请求，再继续探索 MCP 工作流。

---

## 完成第一个 HTTP 请求

ZenRequest 的主工作台默认就是为“构造请求 → 执行请求 → 查看响应”这个高频闭环准备的。

建议你这样开始：

1. 新建一个请求标签页
2. 选择 HTTP 模式
3. 输入请求 URL
4. 根据需要填写方法、参数、请求头或请求体
5. 点击发送
6. 在响应面板查看结果

在这个阶段，你不需要一次性理解所有编辑能力。先掌握这条主链路最重要：

- URL 与方法决定请求目标
- 参数、请求头、请求体是补充输入
- 响应面板是你确认结果、定位问题和保存调试上下文的主要位置

如果接口依赖变量或鉴权，也可以继续配置环境变量和认证信息，但第一次上手建议先从最简单的可成功请求开始。

## 理解响应、历史与回放

完成第一次请求之后，ZenRequest 的价值就不只是“发请求”了，而是把执行上下文保存下来，方便你之后复用和排查。

你应该重点理解这三个能力：

### 响应查看

响应面板不仅用于查看响应体，也用于理解：

- 请求是否成功
- 返回的是结构化 JSON、HTML 还是错误信息
- MCP 请求的协议结果与运行诊断

### 历史记录

历史记录会保留你最近执行过的请求，方便你：

- 回看上一次是如何调试成功的
- 对比不同尝试之间的差异
- 快速重新打开某次执行上下文

### 回放执行

当你已经有一个有价值的历史条目时，可以直接回放它，而不必从头重新构造。对于排查回归、重复验证或者复现场景，这很重要。

## 开始使用 MCP 工作台

当你需要测试 MCP Server，而不只是普通 HTTP API 时，可以切换到 MCP 模式。

MCP 工作台的定位不是一个“什么都做的通用 MCP 客户端”，而是一个帮助你验证 MCP Server 行为的调试工作台。

在当前版本里，你可以围绕这些核心操作展开：

- `initialize`
- `tools.list`
- `tools.call`
- 已有基础的 `resources` / `prompts` / `roots` / `stdio` 支持

如果你是第一次接触 MCP，最推荐的顺序是：

1. 先做 `initialize`
2. 再做 `tools.list`
3. 最后选择一个工具执行 `tools.call`

这样你会更容易建立对 MCP 会话和工具调用的直觉，而不是一开始就陷入原始协议细节。

## 通过 stdio 连接本地 MCP Server

如果你的 MCP Server 是通过本地进程运行的，而不是通过 HTTP 暴露，那么应使用 `stdio` 方式。

ZenRequest 在产品内已经提供了 `stdio` 首次引导，这里补充完整理解路径。

### 最小成功路径

最简单的 `stdio` 使用路径是：

1. 在 MCP 工作台中把 transport 切换到 `stdio`
2. 填写 `command`
3. 仅在需要时补充 `args`
4. 仅在需要特定工作目录时填写 `cwd`
5. 先执行基础 MCP 操作验证连接是否成立

### 三个关键字段怎么理解

#### `command`

这是用于启动本地 MCP Server 进程的命令。最稳妥的做法是：

- 直接填写你平时在终端中实际会执行的命令
- 不要先追求抽象或封装，先追求可成功启动

#### `args`

这是传给 `command` 的附加参数。只有当你的服务启动命令本身就带参数时才需要填写。

例如：

- `node`
- 参数：`dist/index.js stdio`

#### `cwd`

这是工作目录。只有当服务依赖特定项目目录、配置文件位置或相对路径时才需要填写。

如果你发现服务在终端里能启动，但在工作台中失败，优先检查：

- `command` 是否正确
- `args` 顺序是否和终端中一致
- `cwd` 是否指向了服务真正期望的目录

## HTTP 请求调试

当你把 ZenRequest 当作日常 HTTP 调试工具时，通常会反复使用这些能力：

- URL 与方法编辑
- Query 参数
- 请求头
- Body（JSON、原始文本、表单、二进制）
- 认证信息
- Mock 模板
- 断言测试

建议的使用方式不是一开始把所有配置都填满，而是按“最小可成功请求 → 逐步补充细节”的节奏推进。这样你更容易知道是哪一部分导致失败。

## MCP 工作台

MCP 工作台适合这几类场景：

- 验证 MCP Server 是否能正确完成初始化
- 确认工具列表、工具 schema 和工具输入是否符合预期
- 检查 `resources`、`prompts`、`roots` 等协议面在当前实现中的表现
- 保留历史与回放，重复验证同一组 MCP 操作

使用 MCP 工作台时，建议把“协议理解”和“调用结果验证”分开：

- 先确认操作类型是否选对
- 再确认输入是否符合 server 预期
- 最后通过响应与诊断信息判断问题在连接层、协议层还是业务层

## 导入与工作区整理

ZenRequest 不只是一个临时请求发送器。你可以把已有资产逐步整理进工作区，形成可复用的本地调试环境。

当前主线能力包括：

- 工作区导入 / 导出
- cURL 导入为可编辑请求草稿
- 集合与已保存请求管理
- 环境变量与模板解析

一个高效的做法是：

1. 先把已有 cURL 或请求样例导入
2. 整理成保存请求或集合
3. 配置环境变量，减少重复编辑
4. 通过历史与回放沉淀稳定验证路径

---

## 历史与回放

历史和回放是 ZenRequest 区别于“一次性请求发送器”的关键能力之一。

适合重点使用它们的场景包括：

- 重复验证一个已经成功的请求
- 对比两次不同输入的结果差异
- 复现线上问题时保留本地操作轨迹
- 重放 MCP 调试链路，确认问题是否稳定复现

如果你经常需要重复解释“上次我是怎么成功的”，那么你应该更主动地使用历史与回放，而不是每次重建请求。

## stdio 使用说明

`stdio` 适合本地进程型 MCP Server，但它也比 HTTP 模式更容易受运行环境影响。理解下面这些原则会更稳：

- 先把本地终端中的启动命令跑通，再迁移到工作台
- 不要默认认为所有失败都是协议问题，很多时候是命令路径、参数或工作目录问题
- 先从简单操作验证链路，例如 `initialize`
- 一旦跑通，尽量保留历史和固定配置，减少重复试错

在产品内遇到引导内容时，可以把它理解为“最小成功路径”；而这份手册则帮助你把这条路径扩展成稳定、可复用的工作方式。
