# AGENTS.md

适用于 `/media/qiang/DataDisk/D/MyProject/ZenRequest` 的 AI 编码代理协作说明。

## 1. 目标与定位

ZenRequest 是一个本地优先的桌面 API 工作台，强调以下方向：

- 极速启动
- 轻量运行
- 离线优先
- 隐私优先
- 本地数据持久化

任何实现、重构、测试或文档改动，都应优先服务这些目标。不要为了“通用性”或“未来可能需要”引入与当前产品方向不匹配的复杂设计。

## 2. 项目技术栈

- 前端：Vue 3 + TypeScript + Vite
- 桌面运行时：Tauri 2
- 后端语言：Rust
- HTTP 引擎：`reqwest`
- 本地存储：SQLite + `rusqlite`
- UI：Tailwind CSS + shadcn-vue 风格组件
- 测试：Vitest + Vue Test Utils + jsdom

## 3. 目录与职责划分

优先遵守当前目录边界，不要随意打破模块职责。

- `src/`：前端主代码
- `src/features/`：按业务域拆分的功能模块
- `src/components/`：通用 UI 组件和部分业务组件
- `src/lib/`：跨模块工具、运行时桥接、状态辅助函数
- `src/data/`：静态预置数据
- `src-tauri/`：Rust 与 Tauri 侧实现
- `docs/`：设计、计划和基线文档

当前前端已呈现出较明确的分层模式：

- 应用级状态编排集中在 `src/features/app-shell/composables/useAppShell.ts`
- Tauri 交互边界集中在 `src/lib/tauri-client.ts`
- 请求工作区、快照、状态复制等纯逻辑集中在 `src/lib/request-workspace.ts`

新增逻辑时，优先复用这一模式：

- 组件负责展示与事件转发
- 组合式函数负责状态编排
- `lib` 负责纯函数、转换逻辑、运行时桥接
- 模块之间通过清晰边界协作，而不是相互渗透内部细节
- 不要把大量业务逻辑直接塞进 Vue 组件模板或单个超大组件中

## 4. 核心工作原则

### 4.1 增量开发

- 以小步、可验证的方式推进
- 每一步都尽量保持代码可运行、可测试
- 优先局部修正，不做无关的大范围改写

### 4.2 证据优先

- 修改前先阅读现有实现
- 至少参考 3 处相似模式再动手
- 以仓库现状为准，不凭印象硬套外部最佳实践

### 4.3 实用主义

- 优先选择朴素、稳定、易维护的方案
- 简单方案优先于抽象方案
- 没有明确收益时，不新增层级、框架或中间抽象

### 4.4 保持上下文连续

- 后续任务应延续前序命名、结构和模式
- 与当前模块现有实现保持一致
- 相关子任务完成后要验证集成效果

### 4.5 设计与扩展原则

- 涉及新增功能、跨模块改动、数据模型调整、状态流重组或前后端边界变化时，先明确顶层设计，再进入实现
- 设计时优先保证模块边界清晰、职责单一、高内聚、低耦合
- 优先使用模块化和组合式设计，避免把多个变化点耦合在同一实现中
- 抽象必须服务于真实需求或明确的变化点，不得为了假设中的未来需求提前过度设计
- 共享能力、公共接口或中间抽象，原则上应建立在至少两个真实使用场景之上
- 小型改动优先局部修正和最小影响原则，不因为简单需求强行上升到架构层
- 当“可扩展性”和“当前复杂度”冲突时，默认优先选择更简单、可验证、可维护的方案，除非存在明确的近期扩展需求

## 5. 代码质量要求

### 5.1 通用要求

- 遵循项目现有风格和命名方式
- 一个函数或类只做一件事
- 避免重复逻辑，优先抽取已有公共能力
- 公共抽象应建立在真实复用场景之上，而不是提前为未来铺层
- 遵循 YAGNI，不预埋不必要能力
- 保持实现直接、可读、易追踪

### 5.2 前端要求

- 优先遵循现有 Vue 3 组合式 API 写法
- 新状态逻辑优先放入 composable，而不是组件内部堆积
- 保持 props、events、bindings 命名与现有模式一致
- 优先沿用现有 UI 组件和样式 token
- 不要引入破坏现有视觉语言的组件风格

### 5.3 Rust / Tauri 要求

- 保持命令边界清晰
- 前端与 Rust 通信的数据结构要稳定、明确、可序列化
- 错误信息要清晰，但不要泄漏敏感内部细节
- 优先兼容当前本地优先、离线优先的运行模型

## 6. 测试要求

- 所有公开函数和主要交互路径都应有测试覆盖
- 覆盖边界情况、失败路径和错误处理
- 外部依赖应隔离或模拟
- 目标覆盖率为 80% 以上，至少不要降低关键模块信心

修改后，按任务影响范围选择验证：

- 前端测试：`pnpm test`
- 前端构建：`pnpm build`
- Tauri 桌面联调：`pnpm tauri dev`
- Rust 侧检查：`cargo check --manifest-path src-tauri/Cargo.toml`

如果没有运行验证，必须明确说明未验证以及原因。

## 7. 错误处理要求

- 对可预期失败使用明确的错误处理路径
- 错误文案应帮助定位问题
- 优先降级处理，不要轻易让整个流程崩溃
- 不暴露凭证、令牌、内部实现细节或敏感路径

## 8. 上下文获取与调研

开始实现前先收集上下文，不要盲改。

优先顺序：

1. 先看仓库规范文件和 README
2. 再看目标模块的入口与相邻实现
3. 再找至少 3 处相似模式
4. 最后再落代码

推荐优先阅读：

- `README.md`
- `package.json`

- `src/features/app-shell/composables/useAppShell.ts`
- `src/lib/tauri-client.ts`
- `src/lib/request-workspace.ts`

若环境里有更高层的仓库搜索或语义搜索工具，应优先使用这些工具，而不是直接依赖低层 shell 搜索。

## 9. 编辑规则

- 常规文本修改优先使用 `apply_patch`
- 小范围替换可使用 `sed`
- 除非必要，不要使用脚本批量重写文件
- 优先使用 ASCII
- 仅在文件本身已使用或确有必要时引入非 ASCII 内容
- 注释应简洁，只解释不明显的意图或复杂逻辑

## 10. Git 与工作区安全

### 10.1 基本规则

- 只处理当前任务相关文件
- 不要回退不属于当前任务的更改
- 不要执行破坏性 Git 操作，除非用户明确要求
- 提交前只暂存本任务产出的文件
- 使用 `git add <具体文件>`，不要使用 `git add .`

### 10.2 多工具并行协作

这是硬性规则：

- 把现有未提交改动视为其他工具或用户的有效工作
- 如果你的任务与这些改动冲突，立即停止并报告冲突
- 不要覆盖、回滚、重排别人的在途修改
- 即使文件是你也改过的，也必须先阅读并兼容现状，而不是强行恢复旧版本

## 11. 实施流程

### 11.1 开始前

- 明确任务目标和交付物
- 确认影响范围
- 阅读相关文档和现有代码
- 找到至少 3 处相似实现
- 识别当前工作区是否已有冲突改动

### 11.2 开发中

- 保持改动最小化
- 与现有模式对齐
- 同步补测试或更新测试
- 每个关键步骤后做局部验证

### 11.3 完成后

- 运行相关测试或构建
- 确认无明显行为回归
- 检查是否误改无关文件
- 记录未验证项、风险点和已知限制

## 12. 提交与说明要求

向用户汇报结果时，优先说明：

- 改了什么
- 为什么这样改
- 做了哪些验证
- 哪些地方还没验证
- 是否存在风险或后续建议

不要把“理论上应该可以”当成“已经完成验证”。

## 13. 禁止事项

- 不要为了重构而重构
- 不要无理由引入新依赖
- 不要擅自修改无关模块
- 不要在未确认模式前大面积重写
- 不要跳过验证却声称已经修复
- 不要覆盖已有未提交改动
- 不要使用 `git reset --hard`、`git checkout --` 等破坏性命令，除非用户明确要求

## 14. 一句话准则

先理解，再复用；先对齐，再扩展；先验证，再宣称完成。
