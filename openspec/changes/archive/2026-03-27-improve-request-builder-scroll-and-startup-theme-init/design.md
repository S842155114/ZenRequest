## Context

当前请求工作台已经把请求标签、URL 操作栏和参数编辑区拆成了独立组件，但请求参数区的纵向空间边界仍不够明确。`RequestParams` 内部虽然局部使用了 `ScrollArea`，可是在真实长内容场景下，用户仍会感知到上半区内容向下挤压到响应区，说明请求构建区缺少一个稳定的独立滚动边界。

启动阶段也存在类似的接管边界问题。`index.html` 首帧只按系统主题设置 `data-startup-theme`，而运行时真正的 `themeMode` 要等 Vue 启动并完成 bootstrap 后才应用；同时静态启动屏在 `runStartupBootstrap()` 一开始就被移除，导致用户会先看到一个主题、再看到另一个主题，形成明显的浅深色闪烁。

这次变更只涉及前端工作台布局、启动首帧主题初始化和相关测试，不改变运行时 bootstrap 接口、持久化数据结构或后端契约。

## Goals / Non-Goals

**Goals:**
- 让请求工作台在长参数配置场景下保持稳定结构，固定顶部标签栏和 URL 操作栏，仅让请求构建内容区独立滚动。
- 保证请求构建区和响应区之间的视觉边界稳定，不再出现“参数被响应区遮挡”或需要整页滚动才能继续编辑的体验。
- 在 `index.html` 首帧就根据已保存的 `themeMode` 解析主题，并让静态启动屏、应用接管后的启动屏与最终工作台保持同一主题连续性。
- 将静态启动屏的移除时机延后到运行时主题已经生效且应用完成接管之后，避免主题闪烁。

**Non-Goals:**
- 不调整请求与响应面板的业务语义、持久化规则或发送流程。
- 不重构 Tauri bootstrap 协议，也不新增原生启动窗口。
- 不在本次变更中处理启动性能优化本身，重点是布局边界与感知稳定性。

## Decisions

### Decision: 由请求构建主体独占纵向滚动，而不是让整个上半区滚动

`RequestPanel` 继续保留顶部请求标签栏和 `RequestUrlBar` 作为固定的工作台命令面。`RequestParams` 将作为请求构建区唯一的纵向滚动宿主，外层容器补齐 `flex-1`、`min-h-0`、`overflow-hidden` 一致性，确保长内容只在参数构建主体内部滚动。

Rationale:
- 用户的问题不是单个表格没有滚动条，而是请求构建区缺少一个统一的滚动边界。
- 顶部标签栏和 URL 操作栏承担切换、发送、保存和上下文识别职责，滚出视野会降低高频操作效率。
- 让整个上半区或整页滚动会破坏请求区与响应区的固定分工，反而加重“内容互相覆盖”的感知。

Alternatives considered:
- 让整个请求面板滚动。Rejected，因为顶部命令面会随内容一起滚走。
- 只保留各个子区块内部零散滚动。Rejected，因为无法解决请求构建区整体边界不稳定的问题。

### Decision: 在静态文档阶段直接解析持久化主题模式

`index.html` 将直接读取本地持久化的工作区快照，提取 `themeMode`，并在首帧同时设置 `document.documentElement.dataset.theme` 与 `data-startup-theme`。如果存储值为 `system`，则在静态脚本中使用 `prefers-color-scheme` 解析为实际浅色或深色。

Rationale:
- 首帧主题必须在 Vue 挂载前决定，否则运行时再设置只会产生补丁式闪烁。
- 启动屏样式已经依赖 `data-startup-theme`，同步写入 `data-theme` 可以让应用接管前后的主题来源一致。
- 直接读取已有快照即可完成主题预设，不需要等待 bootstrap 返回。

Alternatives considered:
- 继续使用系统主题作为静态首帧默认值。Rejected，因为用户显式选择浅色或深色后，重启仍会闪烁。
- 在 `main.ts` 或 `App.vue` 中更早应用主题。Rejected，因为它们都晚于 HTML 首帧绘制。

### Decision: 静态启动屏只在应用完成正确接管后才移除

`runStartupBootstrap()` 不再在开始阶段立即移除静态启动屏。成功路径下，先完成 bootstrap 数据应用和运行时主题同步，再移除静态启动屏并展示工作台；失败路径下，先让应用内启动失败态完成渲染，再移除静态启动屏，避免空窗或错误主题暴露。

Rationale:
- 当前闪烁来自“静态启动屏过早消失 + 运行时主题稍后生效”的时间差。
- 用户需要的是稳定接管，而不是尽早看到一个尚未完成主题同步的界面。
- 失败态也需要平滑接管，否则静态屏会一直遮挡错误反馈。

Alternatives considered:
- 保持当前启动时立刻移除静态屏。Rejected，因为这是当前主题闪烁的直接来源。
- 启动成功前一直保留静态屏且不渲染应用内启动状态。Rejected，因为失败时会遮挡重试与错误信息。

## Risks / Trade-offs

- [Risk] 请求构建区滚动边界调整后，局部 `ScrollArea` 与外层 `flex` 高度可能出现叠加约束问题。 → Mitigation: 统一校正 `RequestPanel`、`RequestParams`、`TabsContent` 与 `ScrollArea` 的 `min-h-0` / `flex-1` 结构，并用组件测试覆盖。
- [Risk] `index.html` 直接解析本地存储会复制一小段主题解析逻辑。 → Mitigation: 将规则保持为最小集合，只解析 `light`、`dark`、`system` 三种合法值，并与运行时 helper 保持同一语义。
- [Risk] 延后移除静态启动屏后，启动失败路径若未正确切换会遮挡失败态。 → Mitigation: 为成功与失败路径都增加测试，明确验证静态屏移除时机和应用内启动屏可见性。

## Migration Plan

这是纯前端变更，无数据迁移。

1. 先调整 OpenSpec 与测试，明确请求构建滚动边界和启动主题连续性的验收标准。
2. 实现请求工作台滚动容器调整，确保上半区固定命令面与下方响应区边界稳定。
3. 实现 `index.html` 首帧主题预设与启动屏移除时机调整。
4. 运行相关组件测试、应用测试和构建验证后合入。

回滚方式简单：恢复当前请求区容器结构与启动屏移除逻辑即可，不涉及持久化数据回滚。

## Open Questions

当前没有额外的产品决策待确认。如果实现中发现 `Tabs`/`ScrollArea` 组合在某些浏览器环境下仍存在高度计算问题，再单独拆出布局兼容性调整变更。
