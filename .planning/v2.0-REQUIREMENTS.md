# v2.0 Requirements: Local Trust & Execution Foundation

**Status:** active milestone draft  
**Date:** 2026-04-14  
**Scope:** next milestone definition after v1.3 archive

---

## Summary

ZenRequest 已完成 v1.3 `MCP Sampling Debugging` 并进入“等待下一 milestone”状态。下一轮不应直接扩展多 server MCP、agent workflow 或生成式产品入口，而应先加固本地优先产品承诺与执行模型边界。

`v2.0 Local Trust & Execution Foundation` 的目标是：在不破坏当前 HTTP + MCP 主链路的前提下，建立更清晰的 execution envelope、state ownership、secret-safe persistence 与 explainable replay 基础，为后续多 server MCP、mixed HTTP + MCP scenario、agent-safe tooling 铺路。

---

## Goals

- 为 HTTP、MCP 和未来 agent-oriented workflows 建立可扩展的共享 execution envelope
- 明确 durable / cached / ephemeral state 的归属和恢复边界
- 提升本地快照与持久化异常时的可诊断性与恢复可信度
- 建立 secret-safe persistence / projection baseline
- 让 replay/history/diagnostics 从“可重放”升级到“可解释”

---

## Active Requirements

### EX-01 — Shared execution envelope with distinct artifact types
ZenRequest 必须定义并采用一个共享 execution envelope，使 HTTP request execution、MCP interaction 与未来 agent run 可以共享生命周期框架，但仍保留各自独立的 artifact payload 与 replay 语义。

**Why this matters**
- `.planning/research/PITFALLS.md` 已明确指出：把 AI / Agent action 当成“just another request”会导致 request model 持续膨胀，history 和 replay 变得混乱。
- 如果这一层不先定义，后续多 server MCP 或 agent workflow 很容易继续污染 `request` DTO 和 app shell 状态。

**Acceptance signals**
- 有明确的 execution envelope 定义或等价 shared contract
- 能区分至少三类执行信息：authored input、resolved execution snapshot、result artifact
- HTTP / MCP / future agent execution 不需要通过不断给 request DTO 追加临时 optional 字段来表达

---

### EX-02 — State ownership must be explicit
ZenRequest 必须明确 durable state、cached state、ephemeral UI state 的边界、来源与恢复优先级，避免 frontend snapshot、runtime cache 与 backend persistence 长期漂移。

**Why this matters**
- `.planning/codebase/CONCERNS.md` 已指出 startup state 当前跨 browser localStorage 与 SQLite 两套系统，恢复优先级脆弱。
- MCP session、tool discovery cache、future agent memory 若继续叠加在现有不清晰边界上，后续会显著增加 bug surface。

**Acceptance signals**
- 有 state ownership map 或等价文档
- startup / restore 规则能解释 browser snapshot 与 Rust/SQLite durable state 的 precedence
- runtime cache 与 durable replay/history artifacts 有明确区隔

---

### LT-01 — Corruption must be diagnosable, not silently swallowed
当 browser snapshot 或 SQLite JSON-backed persisted rows 出现 malformed / corrupted 情况时，ZenRequest 不应继续以静默 default fallback 掩盖问题，而应提供可诊断信息与安全恢复路径。

**Why this matters**
- 当前已知问题包括：
  - local snapshot malformed JSON 时静默 reset
  - request/history/environment rows 中 malformed JSON 被 default fallback 吞掉
- 本地优先产品如果不能解释“状态为什么丢了”，信任会直接受损。

**Acceptance signals**
- corrupted local snapshot 会触发明确诊断或 recoverable status
- malformed persisted JSON rows 可被检测、报告或 repair-guided，而不是只被解释为空/default
- 相关 recovery behavior 有 focused tests 覆盖

---

### LT-02 — Secret-safe persistence baseline
ZenRequest 必须建立 secret-safe persistence baseline，确保 token、password、API key、resolved auth values 等敏感信息不会默认以不加区分的方式进入 browser localStorage、导出工件、replay artifacts 或未来 AI context assembly path。

**Why this matters**
- `.planning/codebase/CONCERNS.md` 已明确指出 secrets 目前会进入 browser local storage。
- `.planning/research/PITFALLS.md` 进一步指出，一旦未来增加 replay/export/AI summarization，这些泄漏会被放大。

**Acceptance signals**
- 有敏感字段 inventory 或等价策略定义
- persistence/projection 至少区分 authoring values、resolved execution values、safe projection values
- secret-bearing fields 默认被 redact、exclude 或 isolate，而不是 blind persistence

---

### DX-01 — Replay must be explainable, not only rerunnable
ZenRequest 的 replay/history/diagnostics 模型必须保留足够元数据，让用户理解某次执行“发生了什么、为什么这样发生、依赖了哪些上下文边界”，而不仅仅是可以重新执行一个 payload。

**Why this matters**
- replay-only history 对 HTTP 勉强可用，但对 MCP session / future agent run 很快会失去 explainability。
- `.planning/research/FEATURES.md` 与 `.planning/research/PITFALLS.md` 都强调 inspectability 和 explainability 才是 agent-era workbench 的核心价值。

**Acceptance signals**
- replay artifact 不只是 request/response blob
- diagnostics/history 至少能反映关键 execution context
- 模型为 future MCP session metadata 与 future approval/intervention metadata 留出了扩展位

---

### AR-01 — Architecture boundaries must be preserved
本 milestone 的实现必须继续遵守当前分层：

`UI → composables/state → domain helpers → tauri-client → commands → services → core/runtime → storage`

不允许把新的 execution policy、protocol logic 或 persistence decisions 回塞进 `.vue` 组件或 app shell glue code。

**Why this matters**
- `.planning/research/ARCHITECTURE.md` 与 `.planning/research/PITFALLS.md` 都把“runtime logic drift into Vue shell”列为关键风险。
- 这类 drift 会直接削弱 headless execution、测试可维护性和未来扩展能力。

**Acceptance signals**
- frontend-to-backend 调用继续统一经过 `src/lib/tauri-client.ts`
- commands 继续保持 thin boundary
- service/core 层承担 execution 与 persistence 逻辑
- 新增行为不依赖 scattered component conditionals 才能成立

---

## Out of Scope

以下内容在 v2.0 中默认不纳入范围，除非后续有强证据要求调整：

- 完整 multi-server MCP management UI
- 完整 agent workflow product surface
- 通用 AI chat pane 作为主入口
- 团队云工作区、账号体系、强协作中心
- 重量级 plugin / extension SDK
- 超前扩展大批新协议
- 大规模 prompt/spec generation 产品化入口

这些方向可以成为后续 milestone 候选，但不应该在当前 foundation milestone 中抢占核心资源。

## Risks and Tradeoffs

### Tradeoff: 这不是最“炫”的 milestone
它更偏 foundation，而不是最适合做 demo 的 feature bundle。

### Why still worth it
- 能降低未来 multi-server MCP / agent workflow 的返工成本
- 能保护本地优先、可回放、可检查的产品承诺
- 能让后续 roadmap 建立在更清晰的 execution/data model 上

### Main risk
如果 milestone 过度抽象化，就会变成内部整理而缺乏用户价值。

### Mitigation
每个 phase 都应尽量绑定一个用户能感知的 trust improvement：
- 更清晰的 diagnostics
- 更安全的 persistence
- 更可解释的 replay
- 更可信的 recovery 行为
